### 🕵️ 悬案一：Node 1 (Mode A) 为什么会触发 `assertion failed`？

在 22 号的测试中，你的 Node 1 报错：
`qemu_mutex_lock_iothread_impl: assertion failed: (!qemu_mutex_iothread_locked())`

这个断言的意思是：“**你试图给 QEMU 核心上锁（BQL），但我发现你早就已经上过锁了！**”（双重死锁）。

**元凶在哪里？**
在 `Gemini.txt` 的 `wavevm-cpu.c` 文件中，我们看 `wavevm_remote_exec` 函数的 Kernel 分支。当远程执行完毕，返回 IO 中断结果时，代码是这么写的：
```c
            if (kctx->exit_reason == KVM_EXIT_IO) {
                // ...
                qemu_mutex_lock_iothread(); // <--- 致命 Bug：多余的加锁！
                wavevm_handle_io(cpu);
                qemu_mutex_unlock_iothread();
            } 
```
**物理真相**：在 QEMU 5.2 中，处理内存/IO 的底层函数（如 `address_space_rw`）内部是**自带加锁机制**的。如果在外面强行套一层 `qemu_mutex_lock`，QEMU 的安全引擎就会当场判定双重死锁，直接 `abort()`。
这就是为什么只要 Node 1 试图把 IO 任务甩给远端，一回来就**立刻断言自爆**！

### 🛠️ 终极修正案（基于新版 V30）

请打开你的 **新版（V30）** 的 `wavevm-qemu/accel/wavevm/wavevm-all.c`。

找到 `wavevm_init_machine` 函数。它现在大概长这样（漏了点火）：

```c

    WaveVMAccelState *s = WAVEVM_ACCEL(ms->accelerator);
    // ... (一堆自动探测 mode 的逻辑) ...
    
    // [致命缺失] 这里直接进 wavevm 初始化了，没起 KVM！

    else ret = wavevm_init_machine_user(s, ms);
    // ...
}
```

**请务必把它改成这样（加上引擎点火）：**

```c
static int wavevm_init_machine(MachineState *ms) {
    WaveVMAccelState *s = WAVEVM_ACCEL(ms->accelerator);
    // 1. 自动探测逻辑保持不变
    bool has_wvm_dev = (access("/dev/wavevm", R_OK | W_OK) == 0);
    // ... (保留原本的探测逻辑) ...

    // 2. [补回核心] 必须先初始化底层的 KVM 或 TCG 引擎！
    // 否则 QEMU 的 kvm_state 是空的，后面的 CPU 循环全是虚空执行
    int ret;
    if (access("/dev/kvm", R_OK | W_OK) == 0) {
        ret = kvm_init(ms); // <--- 点火！
    } else {
        ret = tcg_init(ms); // <--- 点火！
    }
    if (ret < 0) return ret;

    // 3. 然后才是 WaveVM 自己的初始化
    memory_listener_register(&wavevm_mem_listener, &address_space_memory);

    if (s->mode == WVM_MODE_KERNEL) ret = wavevm_init_machine_kernel(s, ms);
    else ret = wavevm_init_machine_user(s, ms);
    
    return ret;
}
```
